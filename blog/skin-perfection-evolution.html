The Evolution of Skin Perfection: Custom Themes, Ansible, and Bot Wars

Building a high-end web presence for Skin Perfection required more than just a template. This is the story of how we moved from expensive PHP hosting to a streamlined Ansible-driven infrastructure, while navigating the challenges of custom design and aggressive AI crawlers.

Every project begins with a vision. For Skin Perfection, the goal was a clean, premium aesthetic that reflected their brand's commitment to quality. Instead of reaching for a bloated, off-the-shelf theme, we decided to build something tailored from the ground up.

Crafting a Custom Visual Identity

Skin Perfection works extensively with natural products from brands like Chi, Lakshimi, and Botanical Beauty. Consequently, the design couldn't be just another "premium beauty" template; it had to breathe organic simplicity. We used the "Organic Goodness" theme as a starting point but pushed the boundaries to create a truly bespoke identity. To keep the project maintainable while allowing for deep customization, we implemented a Child Theme architecture. This allowed us to inherit core functionality while overriding specific components and styles without breaking update paths.

brush Frontend Tech Stack

  • Custom Landing Page: A completely bespoke entry point designed for maximum conversion and brand storytelling.
  • WooCommerce Integration: Beyond being a beauty salon, the site features a full-scale webshop powered by WooCommerce, allowing customers to purchase their favorite natural products directly.
  • SCSS Styling: Using SASS for modular, maintainable styles. This allowed us to handle complex layouts and animations with ease.
  • Responsive Architecture: Optimized for everything from mobile devices to high-resolution desktops.

Infrastructure: From PCExtreme to Hetzner

Initially, the website was hosted on PCExtreme. Over time, however, we saw the costs creep up—eventually reaching €20 for a relatively simple PHP server. For a site that requires high performance but doesn't need massive enterprise resources, it was time to rethink the hosting strategy.

As someone very familiar with Docker and Kubernetes, I considered using them—they make deployment pipelines incredibly smooth. However, for a single-site setup, they introduce significant resource overhead that simply doesn't justify the cost. Sometimes a more traditional approach is better. The goal was maximum performance with minimal overhead and high reproducibility.

Enter Trellis & Bedrock

I discovered Roots.io, the organization behind professional WordPress development tools like Bedrock and Trellis. We implemented Bedrock to handle the WordPress installation itself. Unlike standard WordPress, Bedrock allows us to declaratively manage all plugins and their versions from code via Composer, leveraging their own specialized WP Packages repository.

For deployment and provisioning, we used Trellis, which uses Ansible to provide a rock-solid, optimized LEMP stack. One of its standout features is how it handles deployments: it uses a symlink-based atomic deployment strategy. This allows us to roll back instantly if a deployment breaks the site, all while preserving the data in the SQL server.

The resource overhead is incredibly low. We migrated the entire site to the cheapest Hetzner cloud server (at just €5.40 per month), and it runs smoother than ever for a fraction of the previous cost.

Performance Tuning: Memory Over CPU

Initially, the default Trellis configuration was a bit too conservative for our needs. The server had plenty of unused RAM, while the CPU was being pushed to its limits during traffic spikes. We decided to tune the PHP-FPM and OPcache settings to better utilize the available hardware.

First, we significantly increased the php_memory_limit from 96M to 512M. This gave WordPress and WooCommerce the breathing room needed for heavy backend tasks without hitting memory exhaustion errors.

Next, we focused on PHP-FPM workers. These are the child processes that handle individual web requests. If you have too many, the CPU spends all its time "context switching" between them (suffocating the processor); too few, and users have to wait for a worker to become free. We reduced the maximum number of children while increasing the initial pool for better responsiveness.

settings_suggest PHP-FPM Workers
  • pm.max_children: 10 → 6
  • pm.start_servers: 1 → 2
memory OPcache Tuning
  • Memory Consumption: 128M → 256M
  • Max Accelerated Files: 4000 → 10000
  • Interned Strings: 8M → 16M

By bumping php_opcache_max_accelerated_files to 10,000, we ensured that every single file in the WordPress core, WooCommerce, and all active plugins could be cached in memory. This drastically reduces disk I/O and CPU overhead, as the server no longer needs to re-read and re-compile the PHP scripts for every single request.

The Crawler Crisis

Everything was running perfectly until the site started experiencing mysterious slowdowns and occasional outages. There were no spikes in user traffic, so I started digging through the Nginx access logs.

The culprit? AI and search crawlers. Meta, in particular, was being incredibly aggressive. In one instance, we recorded over 11,000 requests between 2:00 AM and 9:00 AM—likely an aggressive retry loop or a poorly configured scraping mechanism.

Initially, they were attempting to scan every possible product variation by iterating through every combination of WooCommerce product filters. We quickly updated the robots.txt to disallow these crawl paths, but even with those restrictions in place, the volume remained high enough to overwhelm the small Hetzner instance. It was effectively a localized DoS attack disguised as indexing.

Defense in Depth

My first line of defense was the server-level firewall (ufw). I manually blocked the most offending IP ranges, but it was a game of whack-a-mole. I needed a more scalable solution.

I integrated Cloudflare as a proxy layer. This provided a suite of sophisticated tools to regulate traffic at the edge:

security Cloudflare Security Stack
  • Rate Limiting: Prevents any single IP from overwhelming the server.
  • Regional Restrictions: Limited traffic to relevant geographic regions.
  • Bot Regulation: Automatically identifies and challenges non-compliant bots.

Beyond the Edge: Monitoring & Reliability

Security is only half the battle. While Cloudflare handles the traffic at the edge, I need to know if the origin server is actually healthy. To ensure high availability and catch issues independently of the edge network, I rely on a separate, self-hosted monitoring stack.

analytics Independent Monitoring
  • Uptime Kuma: A simple, self-hosted monitoring tool deployed in my homelab. It pings the site every minute from an external network and sends instant alerts if any latency or downtime is detected.

Conclusion

The Skin Perfection project highlights the importance of matching technology to the problem. By moving to a declarative, Ansible-managed infrastructure on Hetzner, we achieved better performance and lower costs. By layering Cloudflare's security on top of our custom theme, we built a site that isn't just beautiful, but resilient against the modern landscape of aggressive web crawlers.
