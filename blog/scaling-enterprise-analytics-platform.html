Architecting a High-Scale Data & Analytics Platform for Enterprise Banking

Building the foundation for production-grade AI: From declarative YAML configurations to a multi-layered Internal Developer Platform (IDP) powered by Pulumi.

Transitioning from experimental data science to production-grade analytics is a significant hurdle for any large organization. In the context of a major financial institution, this challenge is amplified by strict regulatory requirements, security constraints, and the need for absolute reliability.

We set out to build a comprehensive production platform for data and analytics designed to bridge this gap. The goal was simple: enable data scientists to move fast while ensuring the infrastructure remains secure, compliant, and easy to manage.

Over several years, that foundation evolved from a model-serving solution into a broader internal developer platform spanning batch execution, low-latency APIs, monitoring, streaming, and generative AI. This retrospective focuses not only on the resulting architecture, but also on the engineering practices and design decisions that allowed it to scale across teams.

A Multi-Platform Ecosystem

While the Batch and Real-Time pillars formed the core of our initial offering, the ecosystem quickly expanded to include specialized platforms that addressed unique organizational needs. These platforms, often merged into our area after their initial inception, brought their own architectural flavors and operational models:

  • Reporting Platform: Offering a "road to production" for PowerBI dashboards, backed by Databricks, Synapse, or ADF. Initially infrastructure-focused, it evolved to provide a streamlined deployment flow where dashboards and their underlying data models are automatically updated via CI/CD pipelines whenever a repository changes.
  • BI App Platform: Dedicated to hosting bespoke web applications for analytics. This stack utilized Azure App Service, Azure Front Door, and Azure Container Registry (ACR), with automated deployment gates to ensure security and consistency for custom front-end tools.
  • Contextual Insights: A specialized platform where the focus was on a widget-based user interface. This allowed users to build highly customized, interactive dashboards from a library of components, all backed by managed infrastructure that handled the underlying data complexity.

Batch Platform: From Jobs to Streaming

The Batch Platform was the first core workload capability. It orchestrates heavy processing jobs with Databricks and Airflow, using data-aware scheduling to manage cross-DAG dependencies. As demand grew, it evolved toward event-driven and streaming patterns that could react to upstream signals without abandoning the cost efficiency of batch processing.

Beyond the Happy Path

Production model delivery is more than scheduling a notebook or exposing an endpoint. The platform also had to support ad-hoc runs, dry runs, repeatable test models, and controlled promotion between environments. A maintained dummy model exercised the entire path—from registration and artifact transfer to infrastructure deployment and execution—so platform changes could be validated before they reached customer workloads.

The batch offering grew beyond its original compute pattern as well. We added optional capabilities such as Neo4j, external storage, and other specialized processing services without forcing every workload to carry their operational cost. This became an important design principle: keep the core path opinionated, but expose extensions through explicit, composable interfaces.

Batch Self-Service and Observability

One of the core tenets of the Batch Platform was to put the user in the driver’s seat without requiring them to be infrastructure experts. A declarative YAML interface lets teams define jobs, schedules, compute, and monitoring settings in version control, while GitOps performs validation, package scanning, ITSM integration, and controlled rollout.

workload.yaml 
clusters:
  default:
    spark_version: 15.4.x-scala2.12
    node_type_id: Standard_D3_v2
    runtime_engine: PHOTON
    autoscale:
      min_workers: 2
      max_workers: 5

tasks:
  predict:
    cluster: default
    timeout_seconds: -1
    max_retries: 0
    python_libraries:
      - risk-scoring-model
    python_wheel_task:
      package_name: risk_scoring_model
      entry_point: predict
      named_parameters: {}
    depends_on: []

airflow:
  schedule: "5 9 * * 1-5"
  tags: [risk, scoring]
  outlets: ["landing/risk/scores"]
  input_dataset_names: ["defined/features/customer", "defined/features/account"]
  input_dataset_trigger_rule: ALL
  max_concurrent_runs: 1

notify:
  on_failure: [team@example.com]
  on_success: [team@example.com]

The configuration describes more than a scheduled notebook: it declares the cluster policy, wheel-based task, execution limits, dependencies, input and output datasets, concurrency, and operational notifications. The platform can therefore generate the Databricks and Airflow resources consistently while preserving the controls required for a production data flow.

For observability, we integrated OpenTelemetry, allowing product developers to send custom log statements directly to their dedicated Application Insights workspace. This empowers teams to manage their own alerts, build custom dashboards, and perform deep log searches without platform-level bottlenecks.

As demand for fresher data grew, the Batch Platform also expanded into Spark Structured Streaming. We made the trade-off explicit: incremental batch processing is generally more cost-efficient but introduces more latency, while continuous processing lowers latency at a higher compute cost. Supporting both lets teams choose an execution model that fits the use case rather than forcing every workload into a real-time design.

Real-Time Serving: Low-Latency Models with Safe Releases

Optimizing for Milliseconds

Before committing to AzureML, we conducted extensive benchmarking to ensure we could meet the bank's strict latency requirements. By right-scaling for transaction volume and optimizing queue lengths and worker counts—assuming a 5ms model execution time—we fine-tuned the serving layer. We implemented stress testing using semaphores to precisely map how different throughput levels impacted P99 latency.

Designing the Real-Time Platform

The real-time story was deliberately shaped around an Azure-native managed-service model. We used Azure Machine Learning Managed Online Endpoints for serving and Bicep for infrastructure definition, which kept the platform aligned with the bank's operational and governance expectations while minimizing custom control-plane complexity.

Architecture decisions also helped keep the responsibilities clear. Python remained the primary language for data-science and model-serving code, while C# was used for platform services and APIs outside that core data-science boundary. Requests were routed through Kong with a broker-style API layer, while model traffic splitting and mirroring were managed declaratively on the Azure ML side. This made it possible to introduce safer rollout patterns without forcing every model team to become infrastructure operators.

A further design choice was to collapse feature lookup and inference into the managed online endpoint itself, so the client could submit a single request and receive both retrieval and prediction in one flow. That simplified the external contract and helped keep the serving layer focused on the use case that mattered most: low-latency, governed inference.

The serving path was supported by a Model Store using MLflow for experiment tracking and model registration. For online feature-store access, benchmarking led us away from Cosmos DB and Redis to Postgres. We observed nearly 5x lower latency for our access patterns, while bulk loading millions of rows took minutes rather than hours. This was a reminder that managed-service choices still need to be tested against the workload's actual latency and ingestion profile.

Declarative Delivery for Live Endpoints

For real-time serving, we built a declarative CLI around AzureML endpoints and deployments. It validates configuration before a change reaches Azure, translates intent into repeatable infrastructure operations, and makes safer rollout patterns available without requiring every model team to understand the AzureML control plane. The same configuration-driven approach later informed the wider platform architecture.

The CLI follows a simple but deliberate validate, plan, apply workflow. Teams keep endpoint definitions, model versions, scoring assets, compute settings, and routing rules in version control; the tool compares that desired state with the live AzureML workspace and shows the proposed endpoint, deployment, and traffic changes before reconciling them. This gave model teams an infrastructure-as-code operating model without turning a production release into a series of portal actions or one-off scripts.

One Repository per Customer or Application

Each customer or application owns a dedicated configuration repository. At its root, reusable compute templates and platform-managed environment metadata are kept separate from endpoint definitions. An endpoint directory contains versioned deployments, their scoring code and Conda environment, tests, and a routing declaration. This creates a small, reviewable boundary around each serving API while preventing model teams from duplicating shared workspace or identity settings.

repository layout 
configuration/
├── templates/compute.yaml
├── platform-internal/metadata_production.yaml
└── endpoints/risk-score/v2/
    ├── routing.yaml
    ├── tests/
    └── deployments/
        ├── stable/
        │   ├── definition.yaml
        │   ├── conda.yaml
        │   └── code/score.py
        └── candidate/...

A deployment descriptor selects a registered model and named compute template; the accompanying runtime and scoring assets are part of the desired state. The CLI hashes those deployment assets, allowing it to distinguish an unchanged deployment from one that requires a controlled update. Routing then declares the production and evaluation topology instead of leaving it to a manual gateway change.

routing.yaml 
traffic:
  stable: 100
mirror:
  candidate: 20

Here, stable returns every production response, while 20% of requests are mirrored to candidate for evaluation without changing the caller’s result. Compute profiles, environment connection details, and package-feed settings remain reusable or platform-managed; application teams concentrate on the endpoint version, model assets, and desired traffic policy.

Its most important responsibility was protecting live traffic. The planner rejected invalid routing—such as traffic targeting an undeclared deployment or percentages that do not form a valid distribution—and the apply flow blocked updates or removals that could interrupt a deployment still serving requests. Instead, safe release patterns became explicit: create and wait for a new deployment, gradually route or mirror traffic, move traffic away from the previous version, then retire it. Readiness checks and autoscaling coordination supported that transition, making progressive releases repeatable rather than dependent on operator judgement.

Security, Traceability, and Reproducibility

In a regulated environment, the developer journey must produce evidence by default. Our delivery controls covered dependency scanning, immutable artifacts, least-privilege permissions, deployment traceability, centralized logging, and reproducible model runs. Model artifacts moved from experimentation into serving through a defined handover rather than an informal copy step, preserving the relationship between source, model version, configuration, approvals, and runtime deployment.

These controls were implemented as reusable pipeline components instead of project-specific checklists. Automated tests, linters, configuration validation, and quality gates made compliance part of the paved road. That approach reduced audit effort while making the secure path the easiest path for developers.

Evolution to an Internal Developer Platform

As the ecosystem grew, the goal was not to replace Batch, Real-Time Serving, Reporting, BI Apps, or Contextual Insights with one oversized platform. Instead, the Internal Developer Platform consolidated the shared developer experience beneath them: self-service capabilities, composable infrastructure, delivery standards, security controls, and operational guardrails. Each platform could retain its domain-specific interface while benefiting from a consistent foundation.

The batch architecture also moved away from the historical assumption that one customer maps to one job. A declarative source of truth can describe multiple jobs and their optional capabilities, while deployment and execution remain separate concerns. This removes the need to rebuild infrastructure merely to run work, improves resource utilization, and lets the platform scale capabilities independently from individual customer lifecycles.

We chose Pulumi as our foundation for three key reasons:

  • Immediate Configuration Parsing: Being able to use a real programming language to parse and validate user configurations directly inside the IaC logic.
  • APIs as Resources: Pulumi allows us to manage high-level APIs (like Databricks clusters or AzureML endpoints) as first-class resources.
  • Stack References: A powerful way to pass outputs between different layers of the infrastructure, making it easy to stack layers together.

The Delivery Pipeline Layers

We structured our delivery into three distinct layers to maintain separation of concerns:

  • Components: The reusable blueprints and Pulumi components.
  • Projects: Specific implementations of those blueprints for different capabilities.
  • Deploy: The actual deployment pipelines containing the stacks (the configuration inputs for specific environments).

Everything is semantically versioned, which allows us to compose environments from stable or experimental releases. This makes it incredibly easy to roll out updates incrementally; a user can choose to run their Dev environment on version 1.2.0 while keeping Production stable on 1.0.0. We also use this to spin up temporary "alpha" spokes to test the latest bleeding-edge features without affecting any existing workloads. We use Renovate to ensure that updates to underlying components are rolled out automatically and safely.

This separation of concerns also shaped the real-time platform itself. We kept serving responsibilities focused inside the managed endpoint layer while delegating batch processing, feature preparation, evidence movement, and the broader supporting data plane to MSP-Batch and GDP. It was an architectural choice that made the platform easier to reason about: not every component had to become a monolith, and ownership boundaries remained explicit.

Monitoring as a Platform Capability

Infrastructure health is only one part of operating analytics in production; teams also need to understand model and data behavior. We introduced model portals where users could inspect logs, create dashboards, and configure alerts without waiting for a platform engineer. We then explored Lakehouse Monitoring as a configurable workspace capability, integrating it with the wider data ecosystem rather than treating monitoring as a standalone tool.

This naturally led to broader lifecycle questions: how do we retain evidence, compare champion and challenger models, trace intermediate datasets, or compose multiple models into a governed chain? Working through these concrete use cases with the data platform teams helped shape storage and metadata boundaries. More recent capabilities such as MLflow 3 are interesting in this context, but only when they solve a validated platform need rather than adding technology for its own sake.

The Rise of GenAI and Agentic Systems

As the landscape shifted towards Generative AI, we introduced a RAG (Retrieval-Augmented Generation) blueprint. In this setup, Databricks handles the heavy lifting of chunking and embedding, while Azure AI Search serves as a passive vector database.

To move away from "spaghetti" implementations—where teams were manually parsing PDFs, hardcoding page ranges, and relying on fragile third-party extraction APIs—we provided standardized building blocks. Users can now configure their entire indexing pipeline declaratively via JSON, ensuring a robust and repeatable process.

The first RAG implementation deliberately separated the configurable indexing pipeline, centrally managed middleware and authentication, and the user interface. A template-based customizable UI provided the long-term path, while a simpler fallback interface reduced dependencies and allowed the MVP to deliver value earlier. This separation also made it possible to improve retrieval and governance without coupling those changes to the chat experience.

For more complex requirements, we expanded the platform to support Azure Container Apps. This allows teams to deploy custom LangChain or LangGraph code, which can be exposed through our Kong gateway. This architectural shift has enabled our users to transition from simple chatbots to fully agentic systems capable of sophisticated reasoning and tool use.

We also investigated an AI Gateway for centralized policy enforcement, routing, observability, and cost controls around model APIs. As with the RAG blueprint, the objective was not to mandate a new service prematurely, but to identify which concerns should be solved once by the platform instead of repeatedly by every AI product team.

The Four-Layer Infrastructure Architecture

To manage the scale of a bank, we organized the physical infrastructure into four hierarchical layers:

  1. The Global Layer: Contains the "Hub" infrastructure—Firewalls, DNS resolvers, Private Endpoints for cross-org data access, and a hub-owned Azure Front Door entry point for public application and API access.
  2. Subscription Vending: For every new project, we automatically vend a subscription populated with a "Spoke" network, Key Vault, Log Analytics, and Azure Container Registry.
  3. PaaS Self-Service: On top of the spoke, product developers choose which PaaS offerings they need (Databricks, AzureML, AI Search, etc.) via a self-service catalog.
  4. Workload Projects: The final layer where the actual code and models live. A Git merge can deploy directly through a Pulumi project, but not every workload is encapsulated that way: some capabilities require platform automation to create a pull request in a dedicated repository, invoke a purpose-built CLI, or start another controlled delivery workflow.

Azure Front Door provides the governed edge for workloads that need to be reachable by users or external clients. Instead of exposing each application independently, it centralizes custom-domain and TLS handling, Web Application Firewall policy, health-aware routing, and global traffic distribution before requests reach the appropriate application backend. Keeping this entry point in the global layer gives product teams a consistent, protected route to their web applications and APIs while the platform retains control over the shared internet-facing boundary.

The Architecture in Practice

These layers establish a clear ownership model across workload projects, shared capability projects, and a protected core spoke. Workloads own their orchestration, jobs, application code, and declarative search configuration. Their delivery is intentionally interface-driven rather than tied to one implementation: Pulumi is used where it fits, while other capabilities can be reconciled through repository pull requests, platform CLIs, or controlled pipelines. Capability projects provide reusable services such as Databricks, Azure AI Search, Azure OpenAI, SQL, and web hosting. The core spoke owns cross-cutting foundations: Entra identities, virtual networking, private endpoints, storage, Key Vault, diagnostics, and security controls.

The resulting design is private by default. Processing, search, model services, and storage communicate through private endpoints in a spoke virtual network, with subnet boundaries, network security groups, and centralized firewall policy governing those paths. Workload identities receive only the permissions they need, while teams receive separate operational and monitoring access. Secrets reach compute through Key Vault-backed integrations rather than appearing in notebooks, applications, or deployment configuration.

The end-to-end data flow follows those boundaries: Airflow orchestrates governed Databricks jobs; outputs and source documents land in managed storage; configured search pipelines create indexes; and RAG applications retrieve relevant context before Azure OpenAI generates a response. Application telemetry, platform diagnostics, and job logs are routed to dedicated monitoring and log-storage paths. This lets individual workloads evolve independently while preserving a consistent security, networking, and observability baseline.

From Platform Components to a Product Operating Model

The next iteration treats the platform as a product rather than a collection of infrastructure components. Its goal is to give capability teams a portal-driven, self-service experience built from paved golden paths: reusable capabilities with clear ownership, supported delivery flows, operational integration, and guardrails that reflect enterprise architecture and Azure Well-Architected principles.

This changes the role of the Internal Developer Platform. Instead of asking every team to interpret network, security, and deployment standards, the portal and its reusable building blocks make approved choices available by default. A shared service-management model then connects platform applications and subapplications to ownership, lifecycle, impact analysis, and operational responsibilities. Common naming, terminology, lifecycle conventions, and centrally maintained dependencies reinforce that experience across teams and repositories.

The target foundation is a dual-hub, hub-and-spoke network with dedicated connectivity, data, and management subscriptions alongside application-owned spokes. General outbound traffic is governed through the firewall, while private endpoint traffic uses private routing and centrally managed DNS. Application spokes normally own their private endpoints; only truly shared services belong in the hub. Monitoring follows the same philosophy: a central Log Analytics workspace per environment, protected by RBAC and private access, gives the platform a consistent operational view without making every workload invent its own telemetry foundation.

Some details of this target architecture are deliberately still being validated—such as environment boundaries, data-platform integration, and private monitoring connectivity. Keeping those choices visible as active architecture work, rather than treating them as settled prematurely, is part of the governance model: decisions are captured as ADRs, supported by evidence, communicated, and revisited as implementation feedback arrives.

From Application Configuration to Reconciliation

The resulting delivery model gives each registered application its own workload-configuration repository. It is the single source of truth for the application’s intended state—not a home for shared platform infrastructure or full application source code. Application metadata sits alongside environment folders, which in turn contain capability-specific configurations such as Databricks jobs, AzureML endpoints, web applications, or containers. Small deployment assets may live with the configuration when a capability needs them, but application code is built elsewhere and referenced by an immutable version.

application configuration repository 
application-config/
├── metadata/
│   └── application.yaml          # Application identity and ownership
├── environments/
│   ├── test/
│   └── production/
│       ├── databricks-jobs/      # Scheduled data and ML workloads
│       ├── azureml-endpoints/    # Real-time model endpoints
│       ├── container-apps/       # Containerized applications
│       └── web-apps/             # Web-facing applications
└── pipelines/
    ├── validate.yaml             # Pull-request checks
    └── deploy.yaml               # Deployment requests after merge

Put simply: metadata identifies the application and its owner; environments records what should run in each target environment; and pipelines validate and request the delivery of approved changes. A team only adds the capability folders it needs, so an application with batch jobs and an API does not have to carry configuration for reporting or containers.

Environments are represented as folders rather than branches, so a reviewer can inspect the complete desired state for each target directly in Git. This intentionally favors fully hydrated, WET configuration over hidden transformation steps: some values may be repeated between environments, but the exact result to be deployed is visible in the change itself. A pull request validates application metadata, configuration, permitted deployment assets, and capability-specific contracts before it can be merged.

After a merge, delivery automation identifies the changed application and environment workloads, then asks the matching platform capability to reconcile them. That capability loads the configuration at the immutable merge commit, runs its own validations and gates, and reports status back to the application repository. This preserves one consistent GitOps experience while allowing each capability to use the deployment mechanism that fits its domain: Pulumi for infrastructure-oriented reconciliation, Databricks Declarative Automation Bundles for Databricks workloads, or a dedicated CLI and other controlled workflows where appropriate.

Architecture Is Also an Operating Model

The platform did not scale through technology alone. We used docs-as-code decision logs to record context, alternatives, ownership, and consequences alongside the systems they affected. What started as a team-level practice eventually expanded to architecture governance across a wider organization, making decisions searchable and their rationale durable.

Reusable components were paired with knowledge transfer, live demonstrations, developer sessions, and incremental ownership. Establishing a new real-time team required codebases and documentation, but also review practices, mentoring, and a shared language for declarative infrastructure and GitOps. These social interfaces matter as much as APIs: a design only becomes a platform capability when other teams can understand, operate, and extend it safely.

Next Steps: Extending the Golden Path

Backstage could become the front door to this operating model. Its software catalog can make workload ownership, documentation, APIs, dependencies, and operational links discoverable in one place. Scaffolder templates could then create a new workload repository with the right configuration structure, CI/CD pipelines, documentation, and ownership metadata already in place. Rather than replacing capability-specific delivery mechanisms, a Backstage action can orchestrate them: creating a pull request, starting a controlled pipeline, or invoking the appropriate platform CLI while giving teams a consistent self-service experience.

The same pattern can extend to workloads that are better suited to Kubernetes than managed endpoints or data services. Teams could keep Helm charts, Kustomize overlays, or other Kubernetes manifests alongside their application configuration, with Argo CD reconciling the approved Git state into a cluster. This is another capability-specific path alongside Pulumi and Databricks Declarative Automation Bundles, not a competing configuration model. The platform would still supply the guardrails—namespace and identity provisioning, network policies, secret integration, image controls, observability, and deployment checks—while Argo CD provides the continuous GitOps reconciliation loop. This adds a container-runtime option to the golden path without forcing every workload away from AzureML, Databricks, or other managed services.

The Pulumi Kubernetes Operator offers a complementary route when a workload needs to manage cloud infrastructure and Kubernetes resources together. It represents a Pulumi stack as a Kubernetes custom resource, then runs and continuously reconciles that stack from within the cluster against its declared source and configuration. In practice, Argo CD can keep the Operator’s stack definition in sync from Git, while the Operator provisions or updates the Azure resources, identities, and Kubernetes resources required by that stack. This preserves Kubernetes-native GitOps without reducing the infrastructure layer to static manifests.

Final Thoughts: Moving Toward Code Governance

By combining the flexibility of Pulumi with a rigid "Golden Path" approach for most developers, we've created a platform that scales. The bank's requirements for security and compliance are met through automated gates, while the product developers get a "cloud-native" experience that abstracts away the complexity of managing a subscription or a virtual network.

However, while we’ve focused heavily on infrastructure, deployment flows, and configuration, the next frontier is the application code itself. We see immense value in initiatives that use Cookiecutter templates to help teams "start right and stay right." Providing standardized scaffolds for the code—not just the infrastructure—is the final piece of the puzzle to ensure consistency and quality across the enterprise.

It’s a reminder that in enterprise engineering, the best platform isn't just about the technology—it's about the interface you provide to your developers.
